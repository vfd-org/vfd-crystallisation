# VFD Explorer v1.2 — Scope

**Status:** Active (opened 2026-04-17, follows v1.1 shipped same day)
**Theme:** Infrastructure. v1 and v1.1 grew prediction coverage.
v1.2 builds the **tools** that make the framework usable, auditable,
and swappable — delivering on the promises made in the original WO
that v1 deferred.

## Why v1.2 is infrastructure, not more predictions

The original WO §4.7 promised a DAG consistency walker: "change a
primitive, see what breaks across every prediction." That promise is
not yet delivered. The WO §4.8 also promised a parameter sweep tool.
Neither exists.

The coverage-matrix audit (2026-04-17) showed that the remaining gaps
(quark masses, W/Z/H, electric charges, magnetic moments, running
sin²θ_W) are **not** blocked on Explorer infrastructure — they are
blocked on *physics work that hasn't been done in the papers yet*.
Adding more registry entries with weak derivations would dilute the
credibility signal.

Better to spend v1.2 on tools that make the existing 20 predictions
go further: provable consistency, visible sensitivity, and
interactive showcase notebooks.

## 1. `vfd consistency` — DAG walker

**What it does:** Walk the dependency graph of primitives and
predictions. When a primitive (shell assignment, eigenvalue, Coxeter
exponent) changes, identify every registered prediction that moves,
and re-score each against experiment.

**Interface:**
```
vfd consistency                          # dump the DAG
vfd consistency --diff                   # compare current vs last-committed primitives
vfd consistency --perturb proton_shells  # flip a primitive, see fallout
```

**Implementation:**
- `src/vfd_core/validation/consistency.py`
- Trace primitives via a lightweight registry
  (`src/vfd_core/registry/dependencies.yaml`) listing what each
  prediction depends on: shell assignments, eigenvalues,
  path-graph stats, etc.
- When perturbed, re-run `vfd verify` and show a diff table:
  which predictions broke, which held.

**Value:** The first real demonstration that the Explorer is a
consistency engine, not just a test runner. Makes the
"triply-determined factor 4" kind of claim machine-checkable.

## 2. `vfd sweep` — integer-sensitivity analysis

**What it does:** Vary an integer invariant over a small range and
report which predictions fail first. Turns "the integers are
overdetermined" from an analytical claim into a quantitative stability
result.

**Interface:**
```
vfd sweep alpha-tree --range 135 139       # sweep 137 and report
vfd sweep consciousness-dof --range 80 95  # sweep 87
vfd sweep proton-nmax --range 3 6          # shell-level sweep
```

**Output:** a table showing, for each perturbed integer, which
predictions still pass and which fail. Useful for showing reviewers
"here's why the value must be exactly X."

**Implementation:**
- `src/vfd_core/sweep/integer_sweep.py`
- Uses `monkeypatch`-style context to temporarily replace a constant,
  re-run `verify_all()`, report diff.

## 3. Paper XXII structural additions

Three structural identities from `papers/paper-xxii/scripts/` that
are currently not registered but are clean and exact:

### 3.1 `mass_eigenvalue_ratio_3_5`
**Claim:** λ_9 / λ_15 = 9/15 = 3/5 = Tr(Y²)/Tr(T₃²) from SU(5).
**Method:** direct ratio from MASS_EIGENVALUES tuple.
**Tolerance:** exact.
**Why register:** this is the source of sin²θ_W = 3/8. Explicitly
showing 3/5 in the registry strengthens the derivation chain.

### 3.2 `hopf_fiber_count`
**Claim:** 600-cell decomposes into exactly 12 Hopf fibers × 10 vertices = 120.
**Method:** fiber decomposition from `scripts/build_600cell.py::find_hopf_fibers`.
**Tolerance:** exact.
**Why register:** this is the geometric source of both the kaon
radius (E₁(C₁₀) = 1/φ²) and the generation count (Z₃ action on fibers).

### 3.3 `coxeter_upper_exponents_product`
**Claim:** The product 17·19·23·29 = 215,441 appears in the
E₈ Coxeter element structure. (Minor but verifiable.)
**Tolerance:** exact.
**Why register:** round out the Coxeter-group structural claims
already in the registry.

## 4. Dirac lattice g-factor — honest lattice-level registration

From `papers/proton-radius/computational-findings.md:107-130`:
the 240-dim Dirac Hamiltonian gives an isolated spin-1/2 band at
E = +2.000 with g ≈ 2.068, anomalous a = 0.034, and
a / [α/(2π)] = 29.3 where 29 = λ_3 + λ_4.

**Approach:** register `dirac_lattice_g_factor_band_E2` with
**zero experimental target** (it's a framework-internal lattice
prediction, not continuum) and notes explaining this is the Dirac
spin-band value, not μ_p. This turns the gaps.md entry into a
registered structural claim with an explicit caveat.

**Why this is honest:** it registers what the framework actually
computes, at the right level (lattice), rather than pretending the
continuum mapping is done. Readers see exactly where the derivation
stops.

## 5. Notebooks in `notebooks/explorer/`

Three notebooks showcasing the library. Each:
- imports from `vfd_core` (no copy-paste math)
- renders cleanly on GitHub and nbviewer
- has clear narrative + figures

### 5.1 `01_geometry_tour.ipynb`
The 600-cell: vertices, adjacency, spectrum, Hopf fibers, integer
mode count = 94. Visualises projections of the 120 vertices.

### 5.2 `02_prediction_walkthrough.ipynb`
Run `verify_all()`, show the table, then drill into three
predictions (proton radius, α⁻¹, form factor) with the derivation
shown inline.

### 5.3 `03_consistency_sweep.ipynb`
Demonstrate the `vfd consistency` walker and `vfd sweep` tool:
flip the proton shell assignment, sweep α⁻¹ tree, show the ripple.

## 6. Non-goals for v1.2

- **No new physics derivations.** Quark masses, W/Z/H, charges from
  integer invariants — all remain on gaps.md until someone writes the
  derivation in a paper/script.
- **No CCRO resume.** Still paused.
- **No Papers XXIII–XXVIII gravity numerics.**

## 7. Definition of done

v1.2 ships when:

1. `vfd consistency` and `vfd sweep` work end-to-end from the CLI
   and produce readable output.
2. ≥ 22 registered predictions (up from 20) including the three
   Paper XXII structural additions.
3. Dirac lattice g-factor registered as a structural lattice prediction.
4. Three notebooks render cleanly in nbviewer.
5. CI still green across all Python versions.
6. `docs/` updated: `gaps.md` reduced where v1.2 closes gaps;
   `architecture.md` updated with the new modules; a new
   `docs/consistency.md` page showing the DAG walker in action.

## 8. Execution order

1. **DAG infrastructure:** `registry/dependencies.yaml` +
   `validation/consistency.py` + CLI wiring.
2. **Sweep tool:** `sweep/integer_sweep.py` + CLI wiring.
3. **Register the three Paper XXII structural identities.**
4. **Register the Dirac lattice g-factor.**
5. **Notebooks:** geometry tour → prediction walkthrough → consistency sweep.
6. **Docs:** update architecture, add consistency page, shrink gaps.
7. **CI smoke:** run tests, verify, mkdocs.
8. **Memory update** + v1.2-results.md.

## 9. Risk register

- **DAG walker scope creep.** If tracing every dependency is too
  hard, start with a coarse-grained "one line per primitive" table
  and refine later.
- **Dirac lattice registration confusion.** Must clearly mark it as
  framework-internal lattice, not continuum. Tolerance 0 (exact) with
  no experimental target.
- **Notebooks require jupyter.** Add `[notebooks]` optional dep
  group to pyproject; don't block CI on them.
- **Sweep tool state mutation.** Must use context managers /
  monkeypatch properly to avoid polluting subsequent `vfd verify`
  runs.
