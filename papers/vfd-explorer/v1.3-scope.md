# VFD Explorer v1.3 — Scope

**Status:** Active (opened 2026-04-17, follows v1.2 same day)
**Theme:** Reviewer tooling. Three features that turn the DAG +
sweep infrastructure into arguments a physicist can actually use.

## Why these three

v1.2 built infrastructure (DAG walker, integer sweep). v1.3
*weaponises* it. Each of these three directly answers a question a
sceptical reviewer would ask:

1. **"What if the shell assignment is wrong?"** → `vfd sweep shell:proton …`
2. **"Did you search the space? How do you know this is the unique fit?"** → `vfd adversarial`
3. **"How did you arrive at this number?"** → `vfd why <prediction_id>`

No new physics derivations. No new primitives. Only tools that
reveal structure the framework already has.

## 1. Shell assignment sweep

Extend `vfd sweep` to recognise `shell:<particle>` primitives and
parse tuple-shaped values like `"2,3,4"`.

**Interface:**
```bash
vfd sweep shell:proton "2,3,4" "2,3,5" "3,4,5" "2,4,6"
```

**Output:** one row per candidate support tuple, showing which
predictions pass / fail.

**Implementation:**
- New context manager `_temporary_shell_assignment(particle, support)`
  in `sweep/shell_sweep.py`, patches
  `vfd_core.shells.assignment.SHELL_ASSIGNMENTS` via a fresh
  `MappingProxyType`.
- Extend `cli.py::_cmd_sweep` to detect `shell:*` primitive ids and
  route to the shell sweeper instead of `run_integer_sweep`.
- Value parser accepts `"2,3,4"` and `"(2,3,4)"` forms.

**Why it matters:** the "change a primitive, see what breaks" promise
made in the original WO is currently delivered for integers only.
This closes that gap for the most important primitive class — shell
assignments.

## 2. Adversarial uniqueness search

Systematically enumerate plausible alternative shell supports for a
particle and report which (if any) also fit experiment within
tolerance. Turns "this is the uniquely correct assignment" from a
claim into a computed fact.

**Interface:**
```bash
vfd adversarial shell proton --max-shell 6 --size 3
vfd adversarial shell proton --size 3 --contiguous   # only contiguous supports
```

**Algorithm (v1.3, kept simple):**
1. Enumerate `itertools.combinations(range(1, max_shell+1), size)` —
   all size-*k* subsets of shells 1..N.
2. For each candidate, temporarily swap in the support and run
   verify_all().
3. Keep candidates where every prediction that depends on this
   particle's shells still passes its registry tolerance.
4. Print:
   - The canonical assignment (should always appear)
   - Any alternatives that also pass
   - Counts: total tried, passed, uniquely failed on specific
     predictions

**Output example (hypothetical):**
```
Adversarial shell search: proton, size 3, max_shell 6
Tried 20 candidates.
Passing candidates (all dependent predictions within tolerance):
  (2, 3, 4)    [canonical]
Failed candidates:
  (1, 2, 3) → proton_charge_radius fails (rel_err 75%)
  (2, 3, 5) → proton_charge_radius fails (rel_err 33%)
  ... [17 others]
Uniqueness: 1 / 20 candidates pass. Canonical assignment is unique.
```

**Why it matters:** the single biggest reviewer-convincer. "We
searched the plausible alternatives; only the canonical
assignment fits" is far stronger than an analytical argument that
this assignment is natural.

**Design notes:**
- The search runs `verify_all()` but only the predictions that
  actually touch this particle's shells are scored for
  pass/fail. Use the DAG to filter.
- Failed candidates are summarised by *which* prediction broke,
  not every prediction.
- Total search size is bounded; for size=3, max_shell=6 it's
  C(6,3) = 20 candidates — fast.

## 3. `vfd why <prediction_id>` — provenance walker

Given a prediction id, print a readable derivation chain: what the
prediction is, what primitives it depends on, the current value of
each primitive, the experimental comparison, and the method string.

**Interface:**
```bash
vfd why proton_charge_radius
```

**Output:**
```
Prediction:   Proton charge radius r_p
Paper:        WO-PROTON-RADIUS
Method:       r_p = Tr(L(P_3)) × λ̄_p = 4 × λ̄_p

Depends on:
  shell:proton       Canonical proton shell support {2, 3, 4}
                     Current value: (2, 3, 4)
                     Source: vfd_core.shells.assignment.SHELL_ASSIGNMENTS

  constant:LBAR_P    Reduced Compton wavelength of the proton (fm)
                     Current value: 0.2103089
                     Source: vfd_core.constants.LBAR_P

Result:
  VFD value:    0.841236 fm
  Experimental: 0.8409 ± 0.0004 fm  (PDG 2022)
  Rel. error:   0.0399%
  σ-margin:     0.84
  Status:       PASS (tolerance 0.1%)

See also:
  Other predictions depending on shell:proton:
    - deuteron_charge_radius
    - proton_electron_mass_ratio
    - proton_magnetic_radius
```

**Implementation:**
- `validation/provenance.py` with `explain(prediction_id)` → string.
- Reads from registry + dependency graph + runs `verify_one()` to
  get the current result.
- Resolves primitive `Current value` from the documented source
  path (simple attribute lookup where possible).
- CLI: `vfd why <id>` wraps it.

## 5. Definition of done

v1.3 ships when:
1. `vfd sweep shell:proton "2,3,4" …` works and shows per-candidate
   pass/fail.
2. `vfd adversarial shell proton --size 3` runs end-to-end and
   reports uniqueness stats.
3. `vfd why proton_charge_radius` prints a readable derivation
   chain.
4. Tests pass: ≥ 3 new tests covering shell sweep, adversarial,
   provenance.
5. `docs/consistency.md` extended with shell sweep + adversarial
   sections. New page `docs/why.md` or a section in consistency.md
   documenting provenance.
6. Main CI green. Site builds clean.

## 7. Risk register

- **Shell patch scope.** Replacing the whole `SHELL_ASSIGNMENTS`
  mapping vs. mutating one entry. Use full replacement for safety
  (no partial-state surprises).
- **Adversarial combinatorial explosion.** Cap search size in CLI
  defaults; for large caps, require an explicit `--confirm` flag.
- **Provenance resolver depth.** v1.3 does one level of "current
  value" lookup; if a primitive source is a Python function, show
  the callable name rather than evaluating. Avoid recursive traces
  in v1.3.
- **Dependency freshness.** Adversarial and shell-sweep share the
  same monkeypatch pattern as `integer_sweep`; if a prediction
  module is added later that uses `from vfd_core.shells.assignment
  import X`, it won't see the patch. Enforce the `from vfd_core.X
  as C` pattern for new predictions (already noted in memory).
